# Contributing to Lucidity

Thanks for the interest. A few notes on the current state.

## Pull requests

**Not currently accepted.** Lucidity is in early solo-led development and the
architecture is changing rapidly (the three-transport model, the agent-mediated
communications layer, and the local daemon are all being designed and
prototyped at the moment). Reviewing external PRs against a shifting target
isn't productive yet.

This will change once the architecture stabilizes. When PRs open up, this
file will be updated and a `good-first-issue` label will start appearing on
the issue tracker.

## Issues are welcome

Bug reports, feature ideas, and design feedback are useful and read.

When filing:

- **Bugs** — please include OS/device, steps to reproduce, what you expected,
  and what happened. Screenshots help.
- **Feature ideas** — describe the use case, not just the feature. "I want X"
  is harder to act on than "When doing Y, I struggle with Z, and X would
  help."
- **Architecture / design feedback** — file as an issue with the `discussion`
  label. Be direct; nuanced disagreement is the point.

## Security issues

Do not file security issues in the public tracker. See [`SECURITY.md`](SECURITY.md).

## Licensing of contributions

When PRs do open up, contributions will be accepted under the license that
applies to the directory they modify (AGPL-3.0 for runtime packages, the
Lucidity Commercial License for product apps, MIT for shared tooling configs).
Contributors will be asked to confirm they have the right to submit their
work under those terms.

## Conduct

Be useful, be specific, be kind. Disagreement is fine; rudeness wastes
everyone's time.
